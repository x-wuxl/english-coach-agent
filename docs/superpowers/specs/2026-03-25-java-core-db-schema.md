## 1. 文档目标

这份文档用于把 `java-core` 涉及的 10 张表正式化到接近 DDL 的程度。

目标是提前明确：
- 表名
- 字段名
- 字段类型
- nullable 规则
- 默认值
- 主键/唯一约束
- 外键关系（逻辑外键或物理外键）
- 索引设计
- JSONB 字段结构建议
- 审计字段规范
- Flyway migration 拆分建议

这份文档仍然**不是最终 SQL 文件**，但已经足够指导后续写 `V1__init_schema.sql` 之类的 migration。

## 2. 建模原则

### 2.1 命名原则
- 表名统一使用 `snake_case`
- 字段名统一使用 `snake_case`
- 时间字段统一使用 `timestamp`
- 日期字段统一使用 `date`
- 状态类字段统一使用 `varchar`
- 可扩展结构统一优先使用 `jsonb`

### 2.2 主键策略
第一版建议统一使用：
- 主键：`bigint generated by default as identity`
- 对外展示编码：用 `*_code` 字段承接

原因：
- Java + MyBatis-Plus 下好处理
- 数据库联表也更顺
- 对外可读标识和内部主键分离

### 2.3 外键策略
第一版建议：
- **逻辑上定义外键关系**
- **物理外键先谨慎使用**

理由：
- MVP 阶段更灵活
- 迁移与数据修复成本更低
- 可以先通过应用层保证一致性

如果你更偏传统强约束，也可以在核心表上加部分物理外键，比如：
- `mastery_state.user_id -> user_profile.id`
- `mastery_state.learning_item_id -> learning_item.id`
- `attempt_log.study_session_id -> study_session.id`

但我建议第一版先以逻辑外键为主。

### 2.4 JSONB 使用原则
`jsonb` 只用于：
- 列表型扩展字段
- 解释性结构字段
- 回写摘要字段

核心计算字段不要躲进 `jsonb`。

## 3. 通用审计字段规范

除特殊日志表外，建议所有主业务表统一带：
- `created_at timestamp not null default current_timestamp`
- `updated_at timestamp not null default current_timestamp`

说明：
- `updated_at` 后续可在应用层维护
- 不急着上数据库 trigger

日志类表如 `attempt_log` 可以只保留 `created_at`，也可以保留 `timestamp + created_at` 双字段，前者代表业务发生时间，后者代表落库时间。

## 4. 表设计详解

## 4.1 `user_profile`

### 4.1.1 用途
存用户画像、学习目标、偏好与当前等级画像。

### 4.1.2 建表草案
```sql
create table user_profile (
    id bigint generated by default as identity primary key,
    user_code varchar(64) not null,
    goal varchar(32) not null,
    sub_goals jsonb not null default '[]'::jsonb,
    daily_minutes int not null,
    study_start_time varchar(5),
    review_time varchar(5),
    overall_level varchar(16),
    vocab_level varchar(16),
    grammar_level varchar(16),
    reading_level varchar(16),
    output_level varchar(16),
    preferred_modes jsonb not null default '[]'::jsonb,
    motivation_style varchar(32),
    fatigue_tolerance varchar(32),
    status varchar(16) not null default 'ACTIVE',
    created_at timestamp not null default current_timestamp,
    updated_at timestamp not null default current_timestamp,
    constraint uk_user_profile_user_code unique (user_code),
    constraint chk_user_profile_daily_minutes check (daily_minutes between 5 and 180)
);
```

### 4.1.3 字段说明
- `user_code`：外部用户标识，后续可与 Telegram/前端用户打通
- `sub_goals`：数组，如 `['daily_conversation', 'work_email']`
- `preferred_modes`：数组，如 `['review_first', 'short_session']`
- `status`：建议值 `ACTIVE` / `INACTIVE`

### 4.1.4 索引建议
```sql
create unique index uk_user_profile_user_code on user_profile(user_code);
```

## 4.2 `learning_item`

### 4.2.1 用途
存受控内容池中的学习项。

### 4.2.2 建表草案
```sql
create table learning_item (
    id bigint generated by default as identity primary key,
    item_code varchar(64) not null,
    type varchar(32) not null,
    content varchar(255) not null,
    meaning_zh varchar(255) not null,
    difficulty int not null,
    theme varchar(64) not null,
    tags jsonb not null default '[]'::jsonb,
    examples jsonb not null default '[]'::jsonb,
    related_item_codes jsonb not null default '[]'::jsonb,
    status varchar(16) not null default 'ACTIVE',
    created_at timestamp not null default current_timestamp,
    updated_at timestamp not null default current_timestamp,
    constraint uk_learning_item_item_code unique (item_code),
    constraint chk_learning_item_difficulty check (difficulty between 1 and 10)
);
```

### 4.2.3 JSONB 结构建议
#### `tags`
```json
["daily_life", "high_frequency", "output_core"]
```

#### `examples`
```json
[
  {
    "en": "I need to book a meeting with the team.",
    "zh": "我需要和团队约个会。"
  }
]
```

#### `related_item_codes`
```json
["phrase_001", "sentence_014"]
```

### 4.2.4 索引建议
```sql
create unique index uk_learning_item_item_code on learning_item(item_code);
create index idx_learning_item_type on learning_item(type);
create index idx_learning_item_theme on learning_item(theme);
create index idx_learning_item_difficulty on learning_item(difficulty);
```

## 4.3 `mastery_state`

### 4.3.1 用途
存用户对学习项的长期掌握状态，是系统最核心的状态表。

### 4.3.2 建表草案
```sql
create table mastery_state (
    id bigint generated by default as identity primary key,
    user_id bigint not null,
    learning_item_id bigint not null,
    seen_count int not null default 0,
    correct_count int not null default 0,
    wrong_count int not null default 0,
    correct_streak int not null default 0,
    recognition_score numeric(5,2) not null default 0,
    recall_score numeric(5,2) not null default 0,
    output_score numeric(5,2) not null default 0,
    memory_strength numeric(5,2) not null default 0,
    forget_risk numeric(5,2) not null default 0,
    last_seen_at timestamp,
    next_review_at timestamp,
    last_mode varchar(32),
    recommended_mode varchar(32),
    status varchar(32) not null default 'NEW',
    created_at timestamp not null default current_timestamp,
    updated_at timestamp not null default current_timestamp,
    constraint uk_mastery_state_user_item unique (user_id, learning_item_id)
);
```

### 4.3.3 字段设计说明
- `recognition_score`：偏识别理解
- `recall_score`：偏回忆能力
- `output_score`：偏输出能力
- `memory_strength`：整体记忆强度
- `forget_risk`：遗忘风险
- `status`：`NEW / LEARNING / REVIEWING / WEAK_OUTPUT / MASTERED / ARCHIVED`

### 4.3.4 索引建议
```sql
create unique index uk_mastery_state_user_item on mastery_state(user_id, learning_item_id);
create index idx_mastery_state_user_next_review on mastery_state(user_id, next_review_at);
create index idx_mastery_state_user_status on mastery_state(user_id, status);
```

### 4.3.5 是否加物理外键
可选：
```sql
-- 可后续再决定是否启用
-- alter table mastery_state add constraint fk_mastery_state_user foreign key (user_id) references user_profile(id);
-- alter table mastery_state add constraint fk_mastery_state_item foreign key (learning_item_id) references learning_item(id);
```

## 6. Flyway migration 拆分建议

建议不要把所有东西都塞进一个巨大的 SQL 文件里。

### 推荐拆分
- `V1__init_core_tables.sql`
  - `user_profile`
  - `learning_item`
  - `mastery_state`
  - `study_session`
  - `attempt_log`

- `V2__init_plan_and_review_tables.sql`
  - `daily_plan_snapshot`
  - `daily_plan_item`
  - `weekly_review_snapshot`

- `V3__init_memory_tables.sql`
  - `session_reflection`
  - `behavior_pattern_summary`

- `V4__seed_learning_items.sql`
  - 初始种子内容

这样好处是：
- 主链路先落
- 记忆回写后补也不会乱
- seed 数据和 schema 分离

## 7. 统一字段约束建议

### 7.1 状态字段
所有 `status` 字段先用 `varchar`，不直接上 PostgreSQL enum。

原因：
- 演进更灵活
- Flyway 修改成本更低
- Java 枚举已经能做第一层约束

### 7.2 分数字段
所有分数字段统一用：
- `numeric(5,2)`

适合：
- `accuracy`
- `completion_rate`
- `recognition_score`
- `recall_score`
- `output_score`
- `memory_strength`
- `forget_risk`
- `confidence`

### 7.3 优先级字段
`priority_score` 建议用：
- `numeric(8,4)`

因为复习优先级计算值更适合保留更多小数位。

## 8. 建议补充的 DDL 细节决策

在真正写 SQL 前，还建议把下面这些点最后确认一遍：

1. 主键是否全部统一 `bigint identity`
2. 是否对核心表加物理外键
3. `user_code / item_code / session_code / plan_code / attempt_code` 的生成策略由谁负责
4. `updated_at` 是否完全由应用层维护
5. JSONB 字段是否需要额外 GIN 索引

### 当前建议
- 主键统一 bigint
- 第一版先不强依赖物理外键
- 各类 `*_code` 由应用层生成
- `updated_at` 先由应用层维护
- JSONB 暂时不加 GIN 索引，等真实查询压力出现再说

## 9. 当前结论

第一版 `java-core` 建议保留以下 10 张表：

1. `user_profile`
2. `learning_item`
3. `mastery_state`
4. `study_session`
5. `attempt_log`
6. `daily_plan_snapshot`
7. `daily_plan_item`
8. `weekly_review_snapshot`
9. `session_reflection`
10. `behavior_pattern_summary`

其中最关键的三张表是：
- `mastery_state`
- `attempt_log`
- `daily_plan_snapshot`

因为它们分别承载：
- 长期掌握状态真相
- 单次行为真相
- 每日学习决策真相
